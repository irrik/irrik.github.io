什么是Java的字节码

严格的讲，Java字节码(ByteCode)仅仅指Java虚拟机执行使用的一类指令，通常被存储在.class文件中。
不同平台，不同CPU的计算机指令有差异，但是Java是一门跨平台的编译型语言，这些差异对上层开发者来说是透明的。
上层开发者只需要将自己的代码编译一次，就可以在不同平台的JVM虚拟机中运行它们。开发者甚至可以用Scala，Kotlin这样的语言编写代码，
只要编译器能够将代码编译成.class文件，都可以在JVM虚拟机中运行。

在这里，P师傅所说的“字节码”可以理解更广义一些–所有能够恢复成一个类并在JVM虚拟机里加载的字节序列，都在探讨范围之内

利用URLClassLoader加载远程class文件

Java的ClassLoader是用来加载字节码文件最基础的方法，在P师傅Java安全漫谈系列开头的反射篇中有提到过

ClassLoader是什么呢？就是一个“加载器”，告诉Java虚拟机如何加载这个类。Java默认的ClassLoader就是根据类名来加载类，
这个类名是类的完整路径，例如java.lang.Rruntime

ClassLoader的概念不是三言两语就能说得清楚的，P师傅也没有深入分析，直接介绍了URLClassLoader这个ClassLoader。
URLClassLoader实际上是平时默认使用的AppClassLoader的父类，所以解释URLClassLoader的工作过程实际就在解释默认的Java类加载器的工作流程。
正常情况下，Java会根据配置项sun.boot.class.path和java.class.path中列举到的基础路径(这些路径是经过处理后的java.net.URL类)来寻找.class文件来加载。这个基础路径又分为三种情况：

  1. URL未以斜杠/结尾，这种情况认为是JAR文件，使用JarLoader来寻找类，即为在jar包中寻找.class文件
  2. URL以斜杠/结尾，且协议名是file,则使用FileLoader来寻找类，即为在本地文件系统中寻找.class文件
  3. URL以斜杠/结尾，且协议名不是file,那么使用最基础的Loader来寻找类。

正常开发通常遇到的都是前面两点，什么时候才会出现使用Loader寻找类的情况呢？当然是非file协议的时候，最常见的就是http协议。

  • findClass的作用是根据基础RUL指定的方式加载类的字节码，就像上面URLClassLoader部分讲到的一样，可能会在本地文件系统，jar包或远程http服务器上读取字节码，然后交给defineClass

利用BCEL ClassLoader加载字节码

在这篇文章第一节中，P师傅说所有能够恢复能一个类并在JVM虚拟机加载的字节序列，都在探讨范围内。所以bcel字节码也在讨论范围内，并且占据着比较重要的地位。BCEL的全名是Apache Commons BCEL，属于Apache Commons项目下的一个子项目，但因为被Apache Xalan所使用，而Apache Xalan又是Java内部对于JAXP的实现，所以BCEL也被包含在了JDK的原生库中(位于com.sun.org.apache.bcel)。关于BCEL的详细介绍，P师傅专门出了一篇文传送门，一定要看这篇分析。根据P师傅的分析，JDK会将BCEL放到自己的代码主要是为了支撑Java XML相关功能。

BCEL去哪了

如果在Java 8u261下执行，会抛出一个异常，原因是com.sun.org.apache.bcel.internal.util.ClassLoader这个类不在了。很遗憾，在Java 8u251以后，BCEL这个安全人员的好伙伴就此离家出走了，不知道何时会归来。所以，之后测试fastjson反序列化，记得这里有个坑。

回到正文，继续介绍BCEL加载字节码。可以通过BCEL提供的两个类Repository和Utility来利用。Repository用于将一个Java Class先转换成原生字节码，这里也可以使用javac来编译java文件生成字节码。Utility用于将原生的字节码转换成BCEL格式的字节码,这里其实在前面的BCEL如何使用部分已经举过例子，刷上去看看就懂了。BCEL ClassLoader在Fastjson等漏洞利用链构造的时候都有被用到，其实这个类和前面的TemplatesImpl都出自同一个第三方库Apache Xalan。但是由于在Java 8u251之后，这个ClassLoader被移除，之后只能且用且珍惜了。
