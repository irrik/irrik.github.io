这段代码和ysoserial中的CC3不同，ysoserial中并没有使用到InvokerTransformer，这是因为ysoserial发布之后，引爆了安全界，开发者们自然就去寻找一种安全的过滤方法。比如SerialKiller这样的工具就诞生了。SerialKiller是一个Java反序列化过滤器，可以通过黑名单和白名单的方式限制反序列化时允许通过的类。在其发布的第一个版本代码中，给出了最初的黑名单：

这个黑名单中。InvokerTransformer赫然在列，这也就意味着CC1和CC6这样使用InvokerTransformer的利用链无法利用了。所以CC3应运而生，CC3的目的很明显，就是为了绕过针对InvokerTransformer的过滤。CC3没有使用InvokerTransformer来调用任意方法，而是使用了另一个类com.sun.org.apache.xalan.internal.xsltc.trax.TrAXFilter，这个类的构造方法中调用了(TransformerImpl) templates.newTransformer()，这样就免去了使用InvokerTransformer来调用newTransformer这一步。

但是，缺少了InvokerTransformer，TrAXFilter的构造方法同样也是无法调用的。这里用到一个新的Transformer，org.apache.commons.collections.functors.InstantiateTransformer。InstantiateTransformer也是一个实现了Transformer接口的类，它的作用就是调用构造方法。

这里的目标就是利用InstantiateTransformer来调用TrAXFilter的构造方法，再利用TrAXFilter构造方法中的templates.newTransformer调用到TemplatesImpl里的字节码。构造的Transformer调用链如下：
